# E2E Testing Instructions

This directory contains Playwright end-to-end tests for the packaged Electron app.

## Structure

```
e2e/
├── playwright.config.ts   # Playwright configuration
├── tsconfig.json          # TypeScript config for tests
├── helpers/
│   └── launch.ts          # App launch and connection helpers
└── tests/
    ├── smoke.spec.ts      # Basic functionality tests
    ├── navigation.spec.ts # Sidebar navigation tests
    ├── search.spec.ts     # Search functionality tests
    ├── filters.spec.ts    # Filter tests
    ├── settings.spec.ts   # Settings panel tests
    └── clear-cache.spec.ts # Cache clearing tests
```

## Prerequisites

Tests run against the **packaged application**, not the dev server.

```bash
# Build the app first
pnpm dist:win    # Windows
pnpm dist:mac    # macOS
pnpm dist:linux  # Linux

# Then run tests
pnpm test:e2e
```

## How It Works

1. **Launch helper** starts packaged app with `--e2e` flag
2. App enables remote debugging on port 19222
3. Playwright connects via Chrome DevTools Protocol (CDP)
4. Tests interact with the actual Electron window

```typescript
// e2e/helpers/launch.ts
export async function launchApp(): Promise<AppInstance> {
  const exePath = getExecutablePath();  // Finds packaged app
  const app = spawn(exePath, ['--e2e']);
  
  // Connect via CDP
  const browser = await chromium.connectOverCDP(`http://localhost:19222`);
  const page = browser.contexts()[0].pages()[0];
  
  return { browser, page, close: () => app.kill() };
}
```

## Writing Tests

```typescript
import { test, expect } from '@playwright/test';
import { launchApp, type AppInstance } from '../helpers/launch';

test.describe('Feature', () => {
  let app: AppInstance;

  test.beforeAll(async () => {
    app = await launchApp();
  });

  test.afterAll(async () => {
    await app.close();
  });

  test('should do something', async () => {
    const { page } = app;
    
    await page.click('[data-testid="button"]');
    await expect(page.locator('.result')).toBeVisible();
  });
});
```

## Test Data Attributes

Use `data-testid` for test selectors:

```tsx
// In component
<button data-testid="install-button">Install</button>

// In test
await page.click('[data-testid="install-button"]');
```

## E2E Mode Detection

App detects E2E mode via flag or env var:

```typescript
// src/main/index.ts
const isE2E = process.env['LBK_E2E'] === '1' || process.argv.includes('--e2e');

if (isE2E) {
  // Enable CDP debugging
  app.commandLine.appendSwitch('remote-debugging-port', '19222');
  // Disable analytics
  // Disable Sentry
}
```

## Common Patterns

### Wait for App Ready

```typescript
test('should load', async () => {
  const { page } = app;
  
  // Wait for main content to load
  await page.waitForSelector('[data-testid="main-content"]');
});
```

### Interact with Modals

```typescript
test('should open settings', async () => {
  const { page } = app;
  
  await page.click('[data-testid="settings-button"]');
  await expect(page.locator('[data-testid="settings-modal"]')).toBeVisible();
});
```

### Check Notifications

```typescript
test('should show toast', async () => {
  const { page } = app;
  
  // Trigger action
  await page.click('[data-testid="action-button"]');
  
  // Verify toast appears
  await expect(page.locator('.toast-notification')).toContainText('Success');
});
```

## CI/CD Considerations

- Tests require display (use Xvfb on Linux)
- May need to increase timeouts for CI environments
- Screenshots saved on failure to `test-results/`

## Tips

1. **Keep tests independent** - Each test should clean up after itself
2. **Use meaningful selectors** - Prefer `data-testid` over CSS classes
3. **Wait for stability** - Don't rely on timing, wait for elements
4. **Test real flows** - Focus on user journeys, not implementation
5. **Parallelize carefully** - App state may conflict between tests
